Ensure execute-plan judgment requests are handled automatically by the agent instead of being exposed to the user unless an explicit escalation is needed.

Problem
- When the execute-plan engine reports a task status like DONE_WITH_CONCERNS and requests a judgment, the agent should respond via `execute_plan_judgment`.
- In the observed failure mode, the raw judgment request was surfaced in the chat, causing the user to intervene when the system should have continued automatically.

Desired behavior
- Treat execute-plan judgment requests as internal control flow, not normal user-facing assistant output.
- The agent should immediately call `execute_plan_judgment` when a judgment is requested.
- Only surface the situation to the user if the chosen action is `escalate`.

Suggested scope
- Add an agent behavior rule: pending execute-plan judgments must be answered automatically unless there is genuine ambiguity requiring escalation.
- Define default judgment heuristics for common cases, e.g.:
  - DONE -> accept
  - DONE_WITH_CONCERNS + minor understood concerns + sufficient verification -> accept
  - DONE_WITH_CONCERNS + likely incorrect/incomplete work -> retry
  - blocked/unsafe -> stop
  - missing product decision / requirement ambiguity -> escalate
- Update the harness/UI so pending judgment prompts are not shown directly to the user before the agent responds.
- Add regression coverage to ensure a pending judgment produces a tool call to `execute_plan_judgment` rather than normal assistant chat output.

Success criteria
- Users are not asked to respond to internal execute-plan judgment prompts.
- Judgment callbacks are resolved automatically by the agent in normal cases.
- Only true escalations are surfaced to the user.
